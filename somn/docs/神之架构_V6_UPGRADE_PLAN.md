### 1.1 扩展方法论

1. **分析776个Claw YAML的`cognitive_dimensions`（认知维度）**——提取高频但未被岗位覆盖的维度
2. **分析776个Claw YAML的`position_department`字段**——识别当前部门归属中缺失的职能域
3. **分析贤者工程的Phase 2编码数据（766条SageCode）**——提取智慧函数(wisdom_function)中未被岗位覆盖的能力

## 第二卷：方向二——Claw岗位100%落实

### 2.1 现状问题

当前776个Claw YAML中，`court_position`字段的填入情况如下：

```yaml
# 大部分Claw的岗位字段为空
court_position: ""      # 未映射
position_name: ""       # 未映射
position_department: "" # 未映射
```

只有少数被明确任命为伯爵/侯爵等高级岗位的贤者Claw有完整映射。这意味着大部分Claw虽然在代码层面通过`department`字段与部门关联，但未建立到具体岗位的映射关系。

### 2.2 Claw岗位映射机制设计

#### 2.2.1 映射规则

每个Claw必须且仅能映射到一个架构岗位，采用以下优先级规则：

**第一优先级（明确任命）**：已被`_court_positions.py`的`assigned_sages`列表任命的贤者，直接使用其岗位ID。

**第二优先级（学派+认知维度匹配）**：
- 根据Claw的`school`字段匹配到对应部门
- 根据Claw的`cognitive_dimensions`匹配到该部门内的具体岗位
- 匹配算法：计算认知维度与岗位`domain`/`suitable_schools`的相似度

**第三优先级（专员兜底）**：无法匹配到管理层岗位的Claw，自动归入对应部门的专员序列（七品专员岗，capacity=999）。

#### 2.2.2 映射数据结构

在Claw YAML中补充以下字段：

```yaml
# V6.0 岗位映射（自动计算+人工确认）
court_position: "WZ_LL_4_03_Z"     # 对应 _court_positions.py 中的 Position.id
position_name: "仪制司·推理引擎（正四品）"
position_department: "礼部"
position_pin: "ZHENG_4PIN"
position_domain: "推理引擎"
position_si_name: "仪制司·推理引擎"
nobility: ""
claw_role: "primary"               # primary=主责 | secondary=协责 | standby=待命
mapping_confidence: 0.92           # 映射置信度（自动计算）
mapping_method: "cognitive_match"  # appointed | cognitive_match | specialist_fallback
```

#### 2.2.3 自动映射算法

```
function auto_map_claw_to_position(claw_yaml, positions_registry):
    # Step 1: 检查明确任命
    for position in positions_registry.all():
        if claw_yaml.name in position.assigned_sages:
            return (position.id, "appointed", 1.0)
    
    # Step 2: 学派匹配部门
    department = claw_department_map(claw_yaml.school)
    
    # Step 3: 认知维度匹配岗位
    best_match = None
    best_score = 0.0
    for position in positions_registry.by_department(department):
        if position.position_type == SPECIALIST:
            continue  # 专员作为兜底
        score = cognitive_similarity(
            claw_yaml.cognitive_dimensions,
            position.domain + " " + " ".join(position.suitable_schools)
        )
        if score > best_score:
            best_score = score
            best_match = position
    
    if best_score >= 0.6 and best_match.capacity > len(best_match.assigned_sages):
        return (best_match.id, "cognitive_match", best_score)
    
    # Step 4: 专员兜底
    specialist = find_specialist(department, claw_yaml.school)
    return (specialist.id, "specialist_fallback", 0.3)
```

#### 2.2.4 一岗多人机制

由于776个Claw远多于管理层岗位（490中管理层约200个），需要支持"一岗多人"：

- **管理层岗位**（一正一副）：`capacity=1`保持不变，但引入`claw_role`区分
  - `primary`：主责Claw，由岗位`assigned_sages[0]`担任
  - `secondary`：协责Claw，辅助主责，参与岗位决策
- **专员岗位**：`capacity=999`，所有匹配到的Claw均归入

#### 2.2.5 批量映射实施计划

1. 编写映射脚本`scripts/map_claws_to_positions.py`
2. 遍历776个Claw YAML，按上述算法生成映射
3. 输出映射报告（含置信度分布、无匹配Claw列表等）
4. 人工审核低置信度映射（<0.6）
5. 批量更新Claw YAML文件的岗位字段

| 管辖范围 | 仅神之架构内的部门工作结果 | Somn全项目所有核心记忆/能力 |
| 存储模型 | 扁平MemoryRecord（id/title/content/grade） | 格子化分类存储（分馆/书架/格位三级） |
| 记忆类型 | 6种来源+7种分类 | 扩展至覆盖所有Somn记忆子系统 |
| 权限模型 | 藏书阁内部人员读写，其他只读 | 分级权限（全系统可读/格子管理员可写/大学士可删） |
| 持久化 | YAML文件（甲级/乙级落盘） | 分区持久化 + 语义索引 |

### 3.2 格子化存储架构

藏书阁V3.0将采用**"分馆→书架→格位"三级格子化模型**：

#### 3.2.1 分馆（Library Wing）—— 一级分类

| 分馆 | 编码 | 存储内容 | 对应Somn子系统 |
|------|------|---------|-------------|
| 贤者分馆 | SAGE | 贤者能力画像/智慧编码/Claw记忆 | 贤者工程P1-P4 |
| 架构分馆 | ARCH | 神之架构决策/调度记录/岗位变动 | 神之架构 |
| 执行分馆 | EXEC | 任务执行记录/结果/ROI数据 | ROITracker/AgentCore |
| 学习分馆 | LEARN | 三层学习模型/策略/经验沉淀 | LearningCoordinator |
| 研究分馆 | RESEARCH | 研究发现/策略/颗粒度评估 | ResearchStrategyEngine |
| 情绪分馆 | EMOTION | 消费情绪/感性决策研究 | EmotionResearchCore |
| 外部分馆 | EXTERNAL | OpenClaw抓取的外部知识 | OpenClaw |
| 用户分馆 | USER | 用户画像/偏好/历史 | UserProfile |

#### 3.2.2 书架（Shelf）—— 二级分类

每个分馆下设多个书架，例如贤者分馆下设：

| 书架 | 存储内容 |
|------|---------|
| sage_profiles | 贤者能力画像（Phase 0深度学习文档摘要） |
| sage_codes | 智慧编码（Phase 2 SageCode） |
| sage_claw_memories | Claw运行时记忆（Phase 4） |
| sage_distillation | 蒸馏文档索引（Phase 1） |
| sage_evaluation | 贤者能力评估与演进记录 |

#### 3.2.3 格位（Cell）—— 最小存储单元

每个格位存储一条记忆，采用统一的`CellRecord`数据结构：

```python
@dataclass
class CellRecord:
    """格子化记忆条目 V3.0"""
    id: str                    # 格位唯一标识：{WING}_{SHELF}_{SEQ:06d}

    shelf: str                 # 书架

    tags: List[str]
    metadata: Dict[str, Any]
    # V3.0新增

```

### 3.4 跨系统记忆桥接

藏书阁V3.0将作为**Somn所有记忆子系统的统一汇聚点**：

```

│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐        │

│(Phase2) │ │(Court) │ │      │ │      │ │Engine│ │Core  │

      │                               │
      ↕                               ↕

└─────────┘                    └──────────────┘
```

**桥接规则**：
1. 各子系统的重要产出**主动汇报**藏书阁（submit_memory扩展接口）
2. 藏书阁定期**巡检**各子系统，发现高价值未汇报记忆则主动收录
3. 藏书阁提供**统一查询接口**，支持按分馆/书架/贤者/岗位/Claw/时间等多维检索
4. 格子间支持**跨域引用**（cross_references），建立记忆间的关联关系

## 第五卷：V6.0 核心原则

### 5.1 设计哲学

1. **贤者驱动**：每一个新岗位必须对应至少3位贤者的能力画像需求
2. **格子为本**：藏书阁的格子化是V6.0的核心架构创新，所有记忆以格子为基本单位
3. **桥接互通**：藏书阁不是独立王国，而是所有子系统的记忆汇聚点和分发枢纽
4. **渐进兼容**：V5.2.0的所有接口在V6.0中保持向后兼容

### 5.2 不变原则

- 七人代表大会4票通过制——不变
- 王爵独裁制——不变
- 实战派标识与权益——不变
- 锦衣卫五级卫从体系——不变
- 藏书阁不受任何团队管理——不变（但管理范围扩大至全系统）
- 贤者工程的五层链路——不变（但Phase 4的Claw需映射到架构岗位）
