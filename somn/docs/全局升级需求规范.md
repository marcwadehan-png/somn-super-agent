# 全局升级需求规范 v1.0

## 一、规范目的

确保所有迭代升级在原文件基础上进行，升级后统一覆盖原文件，避免版本碎片化和历史混乱。

**核心原则**：
- 单一真相源（Single Source of Truth）
- 原地升级，覆盖替代
- 版本可追溯，变更有记录

---

## 二、升级类型定义

### 2.1 原地覆盖升级

**适用场景**：文档/代码的主体内容升级，结构基本保持

**规则**：
1. 直接编辑原文件，不创建新文件
2. 文件名保持不变
3. 版本号在文件内部递增
4. 变更日志记录在文件头部或专用CHANGELOG章节

**示例**：
```
原文件: 价值驱动数据研究体系.md (v1.0)
升级后: 价值驱动数据研究体系.md (v2.0) ← 覆盖原文件
```

### 2.2 归档删除升级

**适用场景**：重大架构重构，内容大幅调整

**规则**：
1. 创建新版本文件（如有需要）
2. 将原文件移动至 `docs/归档/` 或 `docs/history/` 目录
3. 新文件使用规范命名
4. 归档文件保留90天后可清理（除非有特殊价值）

**示例**：
```
原文件: 神之架构_V3.md
操作: 移动至 docs/归档/神之架构_V3_20260423.md
新文件: 神之架构_V4.md
```

## 三、版本控制规范

### 3.1 版本号规则

```
主版本.次版本.修订号

- 主版本 (Major): 重大架构变更，不兼容升级
- 次版本 (Minor): 功能新增，兼容升级
- 修订号 (Patch): Bug修复，小幅优化

示例: v1.0.0 → v1.1.0 → v2.0.0
```

### 3.2 文件头部版本标识

所有升级文件必须在头部包含版本信息：

```markdown
---
title: "文件名"
version: "v2.0.0"           # 当前版本
supersedes: "v1.0.0"        # 被替代的版本
last_updated: "2026-04-23"  # 最后更新时间
upgrade_summary: "核心升级说明" # 一句话升级摘要
---
```

### 3.3 变更日志规范

在文件末尾或专用章节记录变更历史：

```markdown
## 变更日志

### v2.0.0 (2026-04-23)
**升级说明**: [一句话核心升级点]

**新增**:
- [新增内容1]
- [新增内容2]

**优化**:
- [优化内容1]
- [优化内容2]

**删除**:
- [删除内容1]

**Breaking Changes**:
- [不兼容变更说明]

---

### v1.0.0 (2026-01-01)
**初始版本**
```

## 四、升级流程规范

### 4.1 升级前检查清单

```markdown
## 升级前检查

- [ ] 确认升级类型（覆盖/归档）
- [ ] 阅读并理解原文件全部内容
- [ ] 确认新版本的核心变更点
- [ ] 准备变更日志模板
- [ ] 备份原文件（如需要）
- [ ] 确认所有引用该文件的文档需要同步更新
```

### 4.2 升级执行流程

```
┌─────────────────────────────────────────────────────────────┐
│                    升级执行流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 分析原文件                                         │
│  ├── 理解原文件定位和内容                                   │
│  ├── 识别核心模块和关键定义                                  │
│  └── 确定升级范围和方向                                      │
│                                                             │
│  Step 2: 执行升级                                           │
│  ├── 原地覆盖: 直接编辑原文件                               │
│  └── 归档升级: 移动原文件 + 创建新文件                      │
│                                                             │
│  Step 3: 更新版本标识                                       │
│  ├── 更新文件头部元数据                                      │
│  ├── 递增版本号                                             │
│  ├── 记录变更日志                                           │
│  └── 更新最后更新时间                                        │
│                                                             │
│  Step 4: 同步引用更新                                        │
│  ├── 搜索所有引用该文件的文档                                │
│  ├── 更新引用路径（如有文件名变更）                          │
│  └── 更新引用内容（如有重大概念变更）                        │
│                                                             │
│  Step 5: 归档处理（如适用）                                  │
│  ├── 移动原文件至归档目录                                    │
│  └── 验证新文件正常工作                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 升级后验证清单

```markdown
## 升级后验证

- [ ] 新文件语法正确，可正常打开
- [ ] 版本号正确递增
- [ ] 变更日志完整记录
- [ ] 相关引用已同步更新
- [ ] 原文件已正确归档（归档升级场景）
- [ ] 文档搜索索引已更新（如有）
```

## 五、归档管理规范

### 5.1 归档目录结构

```
docs/
├── 当前版本/          # 当前使用的文档
│   ├── 价值驱动数据研究体系.md
│   └── 核心策略速查手册.md
│
└── 归档/
    ├── 2026-04/
    │   ├── 神之架构_V3_20260423.md
    │   └── 神之架构_V3_变更说明.md
    │
    ├── 2026-03/
    │   └── ...
    │
    └── index.md       # 归档索引（可选）
```

### 5.2 归档文件命名规范

```
原文件名_YYYYMMDD.md

示例:
神之架构_V3.md → 神之架构_V3_20260423.md
```

### 5.3 归档索引（可选）

如归档文件较多，可创建索引文档：


## 六、引用同步规范

### 6.1 引用追踪

升级涉及文件名变更时，必须追踪所有引用：

```bash
# 搜索所有引用
grep -r "原文件名" docs/ file/ --include="*.md"
```

### 6.2 引用更新规则

| 变更类型 | 更新要求 |
|---------|---------|
| **文件名变更** | 更新所有引用路径 |
| **概念/定义变更** | 更新引用内容，添加变更说明 |
| **删除内容** | 更新引用，标注"已移除"或"已升级" |
| **新增内容** | 如被引用，添加引用说明 |

### 6.3 引用检查清单

```markdown
## 引用同步检查

- [ ] 搜索关键词: [原文件名/核心概念]
- [ ] 检查结果: [列出所有需要更新的文件]
- [ ] 更新状态:
  - [ ] docs/xxx.md - 已更新
  - [ ] file/yyy.md - 已更新
  - [ ] ...
```

## 七、特殊情况处理

### 7.1 多文件联动升级

**场景**：多个相关文件需要同时升级

**规则**：
1. 创建统一的升级报告文档
2. 在每个文件中添加"关联升级"说明
3. 按依赖顺序依次升级

```markdown
## 关联升级说明

本升级为联动升级，关联文件：
- 价值驱动数据研究体系.md (v1.0 → v2.0)
- 核心策略速查手册.md (v1.0 → v2.0)
- ROI追踪规范.md (新建)

升级顺序: ROI追踪规范 → 价值驱动数据研究体系 → 核心策略速查手册
```

### 7.2 代码与文档联动升级

**场景**：代码变更需要同步更新文档

**规则**：
1. 代码升级优先（代码是真相源）
2. 文档必须同步更新
3. 在代码中注释关联文档路径

```python
# 关联文档: docs/ROI追踪规范.md
# 版本: v1.0.0
class ROITracker:
    ...
```

### 7.3 紧急修复升级

**场景**：紧急Bug修复或安全更新

**规则**：
1. 跳过部分流程，快速覆盖
2. 事后补全变更日志
3. 记录紧急修复说明

```markdown
### v1.0.1 (2026-04-23) [紧急修复]
**修复**: [Bug描述]
**影响**: [影响范围]
**验证**: [验证方法]
```

## 八、违规处理

### 8.1 禁止行为

- **禁止**创建带版本号的新文件同时保留原文件
- **禁止**在同一目录保留多个版本的同一文档
- **禁止**在归档前删除原文件（除非明确是覆盖升级）
- **禁止**升级后不更新变更日志

### 8.2 违规处理流程

```
发现违规 → 发出警告 → 要求整改 → 限期修正 → 强制处理（如未整改）
```

### 8.3 定期审计

每月进行一次文档审计，检查：
- 是否有未归档的旧版本
- 是否有孤立的文档碎片
- 引用是否完整同步
- 变更日志是否完整
