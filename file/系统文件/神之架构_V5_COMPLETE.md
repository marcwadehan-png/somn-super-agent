## 第一卷：V5核心架构——从线性到网状

### 1.1 V4架构的核心问题

V4.2.0的"全贤就位，百官齐备"虽然实现了岗位的全面任命，但在调度层面仍存在以下问题：

**问题一：静态路由依赖**
原有`DEPARTMENT_SCHOOL_MATRIX`采用硬编码方式定义ProblemType→Department的映射关系，无法根据问题的实际复杂度动态调整。当面对简单问题时仍可能触发多部门联合调度，造成资源浪费；面对关键问题时又可能因预设路由不足而遗漏重要协同。

**问题二：部门边界刚性**
各系统（皇家/文治/经济/军政/标准/创新/审核）之间的协同关系被固化在代码中，跨部门协作需要手动配置。新的问题类型出现时，必须修改代码才能建立新的协同路径。

**问题三：主线独立性过强**
7条主线（皇家/文治/经济/军政/标准/创新/审核）各自运作，缺乏横向的信息流通机制。决策链呈现"皇帝→部门→岗位"的树状结构，而非网状协同。

**问题四：反馈机制缺失**
执行结果无法反向影响调度策略。翰林院的审核意见、藏书阁的价值评估、锦衣卫的效能数据都停留在结果记录层面，未参与下一次调度的优化。

### 1.2 V5核心理念：网状串联

V5.0.0针对上述问题，提出"主线网状串联"的新架构：

**创新一：动态复杂度评估**
根据问题类型、上下文关键词自动评估复杂度（SIMPLE/MODERATE/COMPLEX/CRITICAL），动态决定参与调度的主线数量和协同方式。

**创新二：跨线协同矩阵**
建立7条主线之间的网状协同关系（PARALLEL/SEQUENTIAL/FEEDBACK/HYBRID），支持跨部门的动态信息流通。

**创新三：学习优化机制**
记录每次路由选择的结果，通过历史数据优化权重参数，使调度策略随系统运行持续进化。

**创新四：兼容V4接口**
新增网状路由层的同时保留V4接口，确保现有代码不需大规模重构即可迁移。

### 1.3 V5.2.0新增内容

**Claw子系统整合**
贤者Claw已全面入职神之架构，768位贤者对应763个Claw配置，实现完整的智能体协作体系。

### 2.2 跨线协同关系

7条主线之间存在网状协同关系，协同类型分为四种：

**并行协同（PARALLEL）**：多线同时处理不同子任务，结果汇总后输出。适用于问题可分解为独立子任务的情况。

**串联协同（SEQUENTIAL）**：上一线输出作为下一线输入，形成流水线。适用于有明确先后顺序的工作流。

**反馈协同（FEEDBACK）**：执行结果反向影响调度策略。适用于需要根据效果调整方案的场景。

**混合协同（HYBRID）**：并行+串联的组合，根据问题特征自适应选择。

## 第三卷：动态路由算法

### 3.1 问题复杂度评估

V5引入问题复杂度评估机制，根据问题特征自动决定调度规模：

| 复杂度等级 | 指标关键词 | 参与部门上限 | 参与主线上限 |
|------------|-----------|-------------|-------------|
| SIMPLE（简单） | 单一领域、明确目标、标准解法 | 1 | 1 |
| MODERATE（中等） | 多领域交叉、需多视角、创新方案 | 3 | 2 |
| COMPLEX（复杂） | 跨系统问题、利益冲突、战略影响 | 5 | 4 |
| CRITICAL（关键） | 全局影响、生死攸关、变革需求 | 99 | 7 |

### 3.2 动态路由流程

当一个问题进入调度系统时，V5执行以下流程：

**Step 1：复杂度评估**
分析问题类型的关键词，计算复杂度得分，确定复杂度等级。

**Step 2：主线选择**
根据复杂度等级和问题的具体特征，选择参与调度的主线组合：
- SIMPLE：仅主调度主线
- MODERATE：主调度主线 + 1条协同主线
- COMPLEX：主调度主线 + 2-3条相关主线
- CRITICAL：全部7条主线联动

**Step 3：节点构建**
从选定主线的岗位体系中提取关键节点，组成调度路径。每个节点包含：部门、岗位、贤者、学派组合、权重。

**Step 4：跨线边构建**
根据主线间的协同关系，生成跨线协同边（并行/串联/反馈/混合）。

**Step 5：权重计算**
综合各节点的authority值和各协同边的权重，计算路径总权重。

**Step 6：执行与记录**
执行调度并记录结果，用于后续学习优化。

### 3.3 核心算法伪代码

```python
def build_routing_path(problem_type, primary_department, context):
    # 1. 评估复杂度
    complexity = evaluate_complexity(problem_type, context)
    
    # 2. 确定主调度主线
    main_line = department_to_main_line(primary_department)
    
    # 3. 根据复杂度构建路径
    if complexity == SIMPLE:
        nodes = build_single_line_nodes(main_line, primary_department)
    elif complexity == MODERATE:
        secondary_line = select_secondary_line(main_line, problem_type)
        nodes = build_parallel_nodes(main_line, secondary_line, primary_department)
    elif complexity == COMPLEX:
        related_lines = find_related_lines(main_line, problem_type)
        nodes = build_multi_line_nodes(related_lines, primary_department)
    else:  # CRITICAL
        nodes = build_full_network_nodes(all_lines, primary_department)
    
    # 4. 计算权重并返回
    return DynamicRoutingPath(
        problem_type=problem_type,
        nodes=nodes,
        total_weight=sum(n.authority for n in nodes)
    )
```

## 第五卷：学习优化机制

### 5.1 路由历史记录

V5记录每次路由选择的关键信息：
- 问题类型和上下文
- 选择的路径（节点+边）
- 执行结果（成功/失败/部分成功）
- 耗时和质量评分

### 5.2 权重自适应

基于历史数据，V5可以自动调整以下参数：

**协同边权重**：根据跨线协同的实际效果，动态调整MAIN_LINE_MATRIX中的权重系数。

**复杂度阈值**：根据实际执行情况，调整各复杂度等级的判定阈值。

**节点优先级**：根据历史成功率的统计，调整同一主线内不同岗位的默认优先级。

### 5.3 反馈闭环

V5构建完整的反馈闭环：

1. **执行层→审核层**：执行结果提交翰林院审核
2. **审核层→皇家层**：审核意见反馈给七人代表大会
3. **皇家层→调度层**：决策调整影响下次路由选择
4. **调度层→执行层**：优化后的路由重新进入执行
