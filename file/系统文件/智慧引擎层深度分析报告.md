## 一、引擎层总体架构描述

智慧引擎层位于Somn系统的第三层（Engine Layer），承担从知识到决策的核心推理能力。整体架构呈现**三层金字塔**结构：

**顶层（协调层）**
- `SuperWisdomCoordinator` — 智慧融合主协调器，17流派统一调度
- `SuperWisdomEngine` — 超级智慧引擎，贤者能力集成（V5.0）
- `ThinkingModeFusionEngine` — 思维模式融合引擎（12种思维模式）

**中层（专业引擎层）**
- 35个WisdomSchool学派引擎（confucianism、daoism、economics、mingjia、wuxing、complexity等）
- 推理引擎：DeepReasoningEngine、ThinkingMethodFusionEngine、NarrativeIntelligenceEngine
- 研究引擎：ResearchStrategyEngine（~1320行）、ResearchPhaseManager（76.69KB）
- 学习引擎：SuperLearningEngine（模块化v2.0，含5个子模块）
- 情绪引擎：EmotionResearchCore（50.63KB，30交叉点框架）

**底层（知识编码层）**
- `wisdom_encoding_registry.py` — 730贤者智慧编码注册表（v2.0，679KB超大文件）
- `EmotionResearchFramework` — 5维度×6方向=30交叉研究点
- 各学派Core类：提供原则定义与分析方法

## 二、各子模块详细分析

### 2.1 V6新增四学派引擎分析（economics/mingjia/wuxing/complexity）

**EconomicsWisdomEngine**（~230行）
- 接口：solve(problem_type, context)、get_wisdom()、get_all_wisdoms()
- 现状：v1.0基础实现，Domain枚举5个，原则7条，Core和Engine双层架构
- 问题：solve()使用if-elif链，新problem_type需手动扩展；缺少与WisdomDispatcher的集成注册

**MingjiaWisdomEngine**（~180行）
- 接口：solve()、get_wisdom()、get_all_wisdoms()
- 现状：3位贤者（惠施、公孙龙、桓团），3种分析模式
- 问题：get_wisdom()参数为sage_name而非principle_name，接口不一致

**WuxingWisdomEngine**（~286行）
- 接口：solve()、get_wisdom(wisdom_type)、get_all_wisdoms()
- 现状：5五行+2贤者，分析模式5种
- 问题：get_wisdom(wisdom_type)使用字符串"five_elements"/"sages"，其他引擎用principle_name

**ComplexityWisdomEngine**（~201行）
- 接口：solve()、get_wisdom()、get_all_wisdoms()
- 现状：6大原则（涌现/适应性/网络/自组织/敏感/适应度景观），3种analyze方法
- 问题：analyze_adaptive_evolution使用固定索引访问principles，可能越界

### 2.2 推理引擎分析

**DeepReasoningEngine**（`reasoning/deep_reasoning_engine/`）
- 核心：7步推理链，模拟人类深度思考过程
- 问题：路径存在但未确认是否已实现完整逻辑

**ThinkingModeFusionEngine**（~500行，thinking_mode_fusion_engine.py）
- 12种思维模式：SYSTEMATIC/DESIGN/CRITICAL/CREATIVE/LOGICAL/LATERAL/PROBABILISTIC/FIRST_PRINCIPLES/INVERSION等
- 问题1：DESIGN模式重复定义（ThinkingMode中有两个DESIGN）
- 问题2：THINKING_MODE_DEFINITIONS字典中部分步骤包含非标准编码词（如"recognize"、"judge"）
- 问题3：INVERSION模式步骤使用"recognize"拼写错误，应为"recognize"

**NarrativeIntelligenceEngine**（4.61KB）
- 规模：最小的引擎文件之一
- 状态：基础占位实现

**SuperWisdomCoordinator**（~35.44KB）
- 协调17大智慧流派，CoordinationStrategy枚举6种
- 关键方法：analyze()带30s硬超时保护（v10.0 P0修复）
- 问题：_load_wisdom_systems()使用try-import失败静默降级，可能掩盖真实错误

### 2.3 研究引擎分析

**ResearchStrategyEngine**（~1320行）
- 核心：研究发现→策略→验证→演化的7阶段闭环
- 四维分类框架：ImpactTarget×TargetAudience×JourneyStage×Mechanism
- 5枚举+5数据类+完整生命周期
- 问题1：策略命名模板_NAMING_TEMPLATE定义但未在代码中使用
- 问题2：findings_to_strategy()方法缺失实现
- 问题3：策略关系网络（协同/冲突/前置/替代/组合）定义了数据结构但未实现推理逻辑

**ResearchPhaseManager**（76.69KB，~2100+行）
- 四阶段21任务：Phase1(5)+Phase2(5)+Phase3(5)+Phase4(6)
- BaseTaskExecutor抽象基类，21个具体TaskExecutor子类
- 问题1（P2）：每个TaskExecutor子类都有大量硬编码的字典数据（如emotion_quadrants、value_tiers），应抽取为配置或知识库
- 问题2（P2）：execute()方法中异常处理过于简单（只有except Exception），应按错误类型分级
- 问题3（P1）：phase_execute()主循环使用同步串行执行，21个任务全串行可能导致整体超时
- 问题4（P2）：_validate_phase_completion()依赖简单的任务计数，未考虑依赖关系

### 2.4 情绪研究引擎分析

**EmotionResearchCore**（50.63KB，~1500+行）
- 框架：5维度×6方向=30交叉研究点（A1-E6）
- 每个交叉点包含：key_frameworks、measurement_tools、application_strategies、risk_factors、neural_mechanisms、ai_applications、sss_wisdom_refs
- Singleton模式+线程锁双检查实现
- 问题1（P2）：框架初始化全部在内存中（约2000行_init_dimension_*），应支持持久化配置
- 问题2（P2）：web_learning_config配置存在但learn_from_web()未实现
- 问题3（P1）：框架中的Somn智慧引用（如"孔子:仁礼之道"）是硬编码字符串，与实际WisdomSchool引擎无动态关联

### 2.5 学习引擎分析

**SuperLearningEngine**（14.78KB，5个子模块）
- 子模块：_sle_types.py(4.15KB)、_sle_engines.py(2.92KB)、_sle_routing.py(10.03KB)、_sle_narrative.py(5.01KB)
- 特性：ThreadPoolExecutor并行学习、超时保护(300s)、学习历史统计
- 问题1（P2）：learn()方法只选择第一个可用引擎，未实现智能路由选择
- 问题2（P1）：capability_engine_map使用硬编码映射，新增引擎需手动维护
- 问题3（P2）：executor线程池在close()未正确shutdown，可能导致资源泄漏

### 2.6 编码注册表分析

**wisdom_encoding_registry.py**（679KB，~9300+行）
- 730位贤者编码（LIT_001-LIT_730+，8大学派）
- 6大认知维度评分（CognitiveDimension枚举）
- 10条通用智慧法则（WL_001-WL_010）
- 问题1（P0）：9300+行单文件，Python解析和IDE加载严重性能问题
- 问题2（P0）：LIT_SAGES列表包含大量重复的core_methods（"知行合一，以知促行"在多个人物中重复出现）
- 问题3（P1）：COGNITIVE_DIMENSION_SCORES定义在SageCode内，但WisdomEncodingRegistry作为独立模块时未导出
- 问题4（P2）：LIT_010等人存在不完整的wisdom_functions（如"express_emotional_depth"缺少前缀）
- 拆分建议：按学派分文件（LIT_sages.py/GOV_sages.py等），主文件仅保留枚举和常量

### 2.7 35学派引擎接口一致性分析

通过grep扫描342个solve/analyze/get_wisdom/process方法签名，发现以下模式：

**接口命名不一致**（P2）
- 主流接口：`solve(problem_type, context)`（约40个引擎）
- 次流接口：`analyze(problem, context)`（约20个引擎）
- 变体接口：`get_wisdom(principle_name)` vs `get_wisdom(sage_name)` vs `get_wisdom(wisdom_type)`
- 建议：统一为`analyze(input, context)`作为主接口，其他作为别名

**Core+Engine双层架构**（P2）
- 约80%引擎采用Core+Engine双层设计
- Core存原则/数据，Engine提供solve()方法
- 问题：部分引擎Core被实例化但Engine的方法直接操作Core数据，双层形同虚设

**占位方法**（P1）
- `dao_wisdom_core.py`仅1.04KB
- `neuron_wisdom_network.py`仅1.02KB
- `military_strategy_core.py`仅568B
- `military_strategy_engine.py`仅687B
- 这些极小文件疑似占位实现或待完成

## 三、引擎注册与懒加载机制分析

### 3.1 WisdomDispatcher._ENGINE_TABLE

位置：`dispatcher/wisdom_dispatch/_dispatch_mapping.py`

_engine_table定义了WisdomSchool到引擎类名的映射，但分析发现：

- V6新增4学派（ECONOMICS/MINGJIA/WUXING/COMPLEXITY）**未在ENGINE_TABLE中注册**
- ENGINE_TABLE中约35个学派，但实际engines/目录有更多命名模式不一致的文件
- _get_engine()使用延迟导入，但未缓存失败结果，连续访问会重复尝试导入

### 3.2 engines/__init__.py懒加载机制

- 使用importlib动态加载_MODULES列表中的199个模块
- 失败时静默pass，可能掩盖ImportError
- 问题：psychology.pioneer_fusion在_MODULES中但路径为psychology_pioneer_fusion.py，存在名称差异

## 五、跨引擎调用链分析

### 5.1 理想调用路径

```
AgentCore/SomnCore
  → SuperWisdomCoordinator.analyze()
    → ProblemDomain分类
    → _discover_candidates() [WisdomSchool筛选]
    → WisdomDispatcher._dispatch()
      → 各学派Engine.solve()
        → EngineCore.analyze_*()
```

### 5.2 实际问题

- SuperWisdomCoordinator依赖的子模块（yangming_xinxue_engine等）使用try-import失败静默
- ResearchPhaseManager与EmotionResearchCore存在隐式耦合（BaseTaskExecutor.erc），但ERC是可选的
- SuperWisdomEngine._load_sage_codes()假设COGNITIVE_DIMENSION_SCORES存在，但实际存储在SageCode的__init__中
- 各引擎的result类型（ComplexityWisdomResult/EconomicsWisdomResult等）互不兼容，fusion时需手动转换

## 六、代码重复模式分析

### 6.1 高重复模式

**Core+Engine双层**：约80%引擎采用，内容重复度高  
建议：提取公共基类WisdomCoreBase，包含_init_principles()、get_wisdom()、_validate_problem_type()等通用方法

**analyze方法签名**：大多数analyze_*方法接受(problem, context=None)  
建议：统一为analyze(input: str, context: Optional[Dict]=None) -> WisdomResult

**Result dataclass**：每个引擎定义自己的Result类  
建议：抽取公共基类BaseWisdomResult，包含problem_type/analysis/confidence字段

### 6.2 重复数据

- EmotionResearchCore中A1-E6的neural_mechanisms字段大量重叠（如"前额叶"在多处出现）
- wisdom_encoding_registry中多个LIT_*人物共享相同的6条core_methods

## 七、性能热点分析

### 7.1 编码注册表查找效率

- matches_problem()方法对每个贤者遍历triggers列表
- 730贤者×平均10 triggers = 7300次字符串匹配
- 建议：建立倒排索引（trigger→[sage_ids]），将查找从O(N×M)降至O(1+M)

### 7.2 engines/__init__.py加载性能

- 199个模块全部在import时尝试加载
- 即使静默失败，importlib.import_module也有开销
- 建议：改为完全懒加载（首次访问时导入）

### 7.3 ResearchPhaseManager内存占用

- 21个TaskExecutor子类实例+每个execute()产生的output_data字典
- 长期运行可能导致内存累积
- 建议：增加结果持久化和内存监控
