# 神之架构 v3.3 行政之鞭制度

> 全局效能双向驱动的强制督促机制
> 版本：v2.0.0 | 日期：2026-04-11
> 执行主体：锦衣卫（厂卫）·效能司

---

## 一、制度总则

### 1.1 制度目的

行政之鞭是神之架构全局效能管理的强制手段，通过**上级施压 + 下级请命**双向驱动模式，实现全层级协同升级：

- **上级施压（鞭策下行）**：上级对下级执行零容错闭环，杜绝怠政躺平，未达标则持续加压直至完成
- **下级请命（效能上促）**：下级对上级逆向驱动，请求资源倾斜与决策加速，强制上级爆发更高效能
- **闭环驱动**：形成"上级施压→下级执行→下级请命→上级效能提升→上级更强压→下级更高效"的正循环

### 1.2 适用范围

本制度覆盖**侯爵及以下全员**，按品秩分为三个鞭策层级：

| 鞭策层级 | 品秩范围 | 施压方向 | 受压方向 |
|---------|---------|---------|---------|
| **高压层** | 侯爵/伯爵/正一至三品 | 七人代表大会 | 各系统首脑 |
| **中压层** | 正四品至从七品 | 各系统首脑 | 各司/处正副长官 |
| **基层** | 正八品至从九品/专员 | 各司/处长官 | 一线专员与执行人员 |

**不适用**：皇帝（最高权）与七人代表大会成员（决策层）——决策层由皇帝直接考核，不纳入行政之鞭。

### 1.3 执行主体

**锦衣卫（厂卫）**是行政鞭子的唯一执行机构，具体由其下设三个司分工：

| 司 | 鞭子职能 | 职责 |
|---|---------|------|
| 监控司 | **效能监察** | 全系统效能数据采集、达标率统计、瓶颈定位 |
| 暗察司 | **怠政侦查** | 检测怠政、拖延、低效行为，生成鞭策指令 |
| 合规司 | **鞭策执行** | 发放鞭子（施压令/请命令）、跟踪整改、闭环验证 |
| **效能司** | **行政之鞭** | **执行全局效能双向鞭策，将大秦指标评分转化为信号灯并执行** |

### 1.4 核心逻辑

行政鞭子以OKR数据为输入，以效能提升为目标，形成闭环：

```
                    ┌─────────────────────┐
                    │   锦衣卫·效能监察    │
                    │  （采集OKR进度数据）  │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   效能评估与分级      │
                    │  绿灯/黄灯/红灯/黑灯  │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
    ┌─────────▼──────┐ ┌──────▼───────┐ ┌──────▼─────────┐
    │ 绿灯·维持运行   │ │ 红灯·施压令  │ │ 黄灯·请命令    │
    │ （无干预）       │ │（上级→下级）  │ │（下级→上级）    │
    └────────────────┘ └──────┬───────┘ └──────┬─────────┘
                               │                │
                    ┌──────────▼────────────────▼──┐
                    │       整改验证与闭环           │
                    │  达标则解除 · 未达标则升级鞭策  │
                    └─────────────────────────────┘
```

## 二、鞭子分级（四级信号灯）

锦衣卫根据OKR进度数据，对每个成员/部门发布四级效能信号：

### 2.1 绿灯（正常运转）

| 条件 | KR进度 >= 目标速度 | OKR综合评分 >= 0.7 |
|------|-------------------|-------------------|
| 鞭策 | 无干预 | 锦衣卫仅记录，不做任何催促 |
| 频率 | — | 每周例行效能快照 |

### 2.2 黄灯（效能预警）

| 条件 | KR进度 0.5~0.8倍目标速度 | 或连续2周进度无变化 |
|------|------------------------|-------------------|
| 鞭策 | **下级请命令** | 下级板块向上级提交资源请求 |
| 内容 | 请求资源倾斜、决策加速、排障支援 | 附具体数据：当前进度、瓶颈原因、所需资源 |
| 频率 | 每48小时一次请命 | 上级须在24小时内响应 |
| 处理 | 上级响应 → 解除黄灯 | 上级未响应 → 升级为橙灯 |

### 2.3 橙灯（效能警告）

| 条件 | KR进度 < 0.5倍目标速度 | 或黄灯持续7天未解除 |
|------|----------------------|-------------------|
| 鞭策 | **上级施压令 + 下级请命令双管齐下** | |
| 上级施压 | 上级板块对下级发布限期整改令 | 明确48小时整改目标 |
| 下级请命 | 下级板块向上级提交紧急资源申请 | 请求双倍资源倾斜 |
| 频率 | 每24小时一轮双向鞭策 | |
| 处理 | 48小时内达标 → 降为黄灯 | 未达标 → 升级为红灯 |

### 2.4 红灯（效能危机）

| 条件 | KR进度 < 0.3倍目标速度 | 或橙灯持续5天未解除 |
|------|----------------------|-------------------|
| 鞭策 | **全面施压令** | 锦衣卫直接介入 |
| 上级施压 | 冻结非核心权限 | 所有资源强制聚焦于目标KR |
| 合规审查 | 锦衣卫合规司启动效能审计 | 检查是否存在怠政/资源浪费 |
| 频率 | 每12小时一轮施压 | |
| 处理 | 72小时内达标 → 降为橙灯 | 未达标 → 升级为黑灯 |
| 后果 | 记入藏书阁效能档案 | 影响下次品秩评定 |

### 2.5 黑灯（效能崩塌）

| 条件 | 红灯持续7天未解除 | 或出现严重怠政行为 |
|------|-------------------|-------------------|
| 鞭策 | **最高级别干预** | |
| 措施 | 锦衣卫上报七人代表大会 | 启动人员替换流程 |
| 合规审查 | 全面效能审计 | 生成崩塌分析报告 |
| 后果 | 品秩降级或岗位替换 | 纳入锦衣卫重点监控名单 |

## 三、双向鞭策机制

### 3.1 鞭策下行（上级→下级·施压令）

**触发条件**：下级大秦指标进度落后于目标速度，由锦衣卫暗察司检测后自动生成施压指令，上级执行。

**施压令内容结构**：

| 字段 | 说明 |
|------|------|
| 目标KR | 需要加速完成的具体关键结果 |
| 当前进度 | 量化数据（百分比） |
| 目标速度 | 应达到的进度速度 |
| 差距分析 | 落后原因、瓶颈定位 |
| 整改时限 | 限时完成节点 |
| 后果警告 | 逾期后果（权限冻结/品秩影响） |
| 资源调配 | 本次施压调配的资源清单 |

**施压强度分级**：

| 等级 | 措施 | 适用 |
|------|------|------|
| 温和提醒 | 书面催办，附进度对比 | 首次黄灯 |
| 限期整改 | 明确48小时整改目标，冻结非核心权限 | 橙灯 |
| 全面施压 | 所有资源强制聚焦，每12小时一次进度检查 | 红灯 |
| 最高干预 | 上报代表大会，启动人员替换流程 | 黑灯 |

**执行原则**：
- 施压必须**附数据**，不允许空话施压
- 施压必须**给资源**，不允许只压不扶
- 施压必须**有时限**，不允许无限施压
- 施压必须**有退路**，达标后立即解除

### 3.2 鞭策上行（下级→上级·请命令）

**触发条件**：下级自身已全力执行但受限于资源/决策/瓶颈，由锦衣卫暗察司检测后生成请命指令，下级提交上级。

**请命令内容结构**：

| 字段 | 说明 |
|------|------|
| 当前状态 | 已完成工作、当前效能数据 |
| 瓶颈描述 | 阻碍提速的具体原因（数据驱动） |
| 资源请求 | 需要上级提供的具体资源/决策/权限 |
| 预期效果 | 资源到位后的效能提升预估 |
| 时效要求 | 请求在多少小时内响应 |

**请命令规则**：
- 下级**必须**在全力执行（绿灯或黄灯连续72小时）后才能提交请命令
- 请命令必须**附效能数据**证明"非怠政而是受限"
- 上级**必须**在24小时内响应请命令（批准/部分批准/拒绝并说明理由）
- 上级未响应 → 锦衣卫自动升级请命为橙灯级别，抄送更上一级

### 3.3 鞭策闭环

每一次鞭策必须形成闭环：

```
鞭策发起 → 执行整改 → 效能验证 → 结果判定
                                          │
                              ┌───────────┴───────────┐
                              │                       │
                          达标·解除               未达标·升级
                          记入藏书阁               下一级鞭策
```

**闭环规则**：
1. 每次鞭策必须**记录至藏书阁**（鞭策令全文 + 整改过程 + 最终结果）
2. 每次闭环必须有**量化验证**（KR进度变化数据）
3. 未达标升级必须**说明理由**（瓶颈分析 + 新的整改方案）
4. 连续三次同一KR被鞭策 → 锦衣卫启动**根因分析**（是否KR设定不合理）

## 四、与OKR体系联动

### 4.1 数据接口

行政鞭子的输入数据完全来自OKR体系：

| 鞭子输入 | OKR数据源 | 采集频率 |
|---------|----------|---------|
| KR进度 | OKRSet.key_results[i].current_value / target_value | 每24小时 |
| 综合评分 | Evaluation.composite_score | 季度评估时 |
| 过程数据 | OKRSet.weekly_updates | 每周 |
| 历史对比 | 藏书阁中的历史效能档案 | 鞭策时按需 |

### 4.2 评分联动

OKR综合评分直接影响鞭策强度：

| OKR综合评分 | 初始鞭策等级 | 说明 |
|------------|------------|------|
| >= 0.7 | 绿灯 | 免鞭 |
| 0.4 ~ 0.7 | 黄灯~橙灯 | 进度关注 |
| < 0.4 | 红灯 | 效能危机 |
| 连续两个季度 < 0.4 | 黑灯 | 效能崩塌 |

### 4.3 品秩联动

鞭策结果影响品秩评定：

| 鞭策记录 | 品秩影响 | 生效周期 |
|---------|---------|---------|
| 季度内无红灯及以上 | 晋升加分 | 季度评定时 |
| 季度内1次红灯 | 不得晋升 | 当前季度 |
| 季度内2次及以上红灯 | 降级预警 | 当前季度 |
| 触发黑灯 | 品秩降级或替换 | 立即 |

## 五、锦衣卫执行规范

### 5.1 鞭策公正性

锦衣卫执行鞭策时必须遵守：

1. **数据说话**：所有鞭策令必须附效能数据，不允许主观臆断
2. **一视同仁**：同品秩、同进度标准下，鞭策强度一致
3. **不偏不倚**：锦衣卫成员自身的效能由七人代表大会监督
4. **申诉通道**：被鞭策者可在24小时内向七人代表大会申诉

### 5.2 鞭策时效

| 鞭策等级 | 响应时限 | 整改时限 | 闭环时限 |
|---------|---------|---------|---------|
| 黄灯 | 48小时 | 7天 | 14天 |
| 橙灯 | 24小时 | 48小时 | 7天 |
| 红灯 | 12小时 | 72小时 | 5天 |
| 黑灯 | 立即 | — | 代表大会裁决 |

### 5.3 鞭策档案

所有鞭策记录纳入**皇家藏书阁·效能档案**，按以下结构存储：

| 字段 | 说明 |
|------|------|
| 鞭策令ID | 唯一标识 |
| 被鞭策者 | 姓名、部门、品秩 |
| 鞭策等级 | 黄/橙/红/黑 |
| KR目标 | 被鞭策的关键结果 |
| 触发数据 | 进度、评分、瓶颈 |
| 整改过程 | 时间线记录 |
| 最终结果 | 达标/未达标/升级 |
| 效能变化 | 鞭策前后的效能对比 |

## 六、特殊保护条款

### 6.1 创新保护

对从事创新探索的成员（皇家科学院、创新专员等）：

- 创新类KR的鞭策阈值**放宽50%**（如标准0.5倍目标速度触发黄灯，创新类KR 0.25倍才触发）
- 创新失败不记入鞭策档案，但**必须有失败复盘报告**
- 连续3次创新失败且无复盘 → 恢复正常鞭策阈值

### 6.2 紧急任务保护

当系统执行皇帝命令（紧急任务）时：

- 所有非相关鞭策自动**暂停**
- 紧急任务相关KR自动进入**绿灯保护期**（不因进度波动触发鞭策）
- 紧急任务完成后恢复常规鞭策

### 6.3 系统级保护

当全系统效能整体下降（非个别成员问题）时：

- 锦衣卫应识别为**系统性问题**而非个别怠政
- 停止对个别成员的鞭策，转而提交**系统效能诊断报告**给七人代表大会
- 代表大会决策后方可恢复鞭策

## 七、制度与其他机制的配合

### 7.1 与锦衣卫合规司的关系

| 机制 | 合规司职责 | 行政之鞭职责 |
|------|----------|------------|
| 合规监督 | 检举违规行为，审查操作合法性 | 督促效能，倒逼执行 |
| 关系 | 合规司是行政之鞭的执行机构 | 行政之鞭是合规司的效能督促职能扩展 |

### 7.2 与翰林院审核的关系

- 翰林院每季度审核**行政鞭子执行报告**，评估鞭策是否公正、合理、有效
- 翰林院有权要求锦衣卫**修正鞭策策略**（如鞭策过于频繁或过于宽松）
- 翰林院审核意见纳入**藏书阁·乙级记忆**（1年审查周期）

### 7.3 与藏书阁的关系

- 所有鞭策档案存储于藏书阁
- 效能档案的保密等级为**丙级**（30天审查周期），同品秩及以上可查阅
- 黑灯记录为**乙级**（1年审查周期），仅高级管理层及以上可查阅

## 九、实施说明

### 9.1 部署步骤

1. 锦衣卫合规司加载行政鞭子引擎（`_whip_engine.py`）
2. 接入OKR引擎（`_okr_engine.py`）的数据接口
3. 初始化全系统效能基线（运行1周采集基线数据）
4. 正式启用鞭策机制

### 9.2 调优周期

- **首月**：观察期，鞭策仅记录不执行（或仅执行黄灯级）
- **次月**：启用黄灯+橙灯，验证双向鞭策效果
- **第三月起**：全四级启用，正式运行
